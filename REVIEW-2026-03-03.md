# Code Quality & Architecture Review — 2026-03-03

## Executive Summary

This is a competent prototype that has grown through disciplined iterative work (3 phases, tracked via AUDIT IDs). The signal architecture, memory system, and LLM fallback design are genuinely well done. However, the codebase is accumulating debt in two critical areas: **agent.gd is becoming a God Object** (677 lines, 7+ responsibilities), and **the core ARCHITECTURE.md proposal — data-driven agents via Resources — was defined but never wired in**, leaving `AgentDefinition` and `PersonalityProfile` as dead code while personality match blocks and hardcoded arrays persist. The project is at a crossroads: either extract the growing complexity into proper systems now, or watch it calcify into something that can't scale beyond 5 agents and 3 zones.

---

## What's Working Well

### 1. Signal architecture is clean and properly decoupled

`agent.gd:2-3` defines two focused signals — `interaction_started` and `state_changed` — that cleanly decouple agent behavior from UI updates. `main.gd:322-323` connects them to UI methods without either side knowing about the other's internals. The `receive_interaction()` pattern (`agent.gd:380-389`) is a good alternative to bidirectional coupling: the initiator calls a well-defined method on the partner rather than reaching into state directly.

### 2. Per-encounter roll tracking (AUDIT-002) is a clever solution

`agent.gd:24` — the `_rolled_for: Dictionary` tracks which agents have been evaluated this proximity window. `_on_area_exited` (`agent.gd:639-642`) cleans up the tracking when agents separate. This solves the original per-frame probability bug elegantly without adding timers or complex state.

### 3. Memory system is well-structured

`memory_service.gd` has good fundamentals:
- Exponential recency decay (`exp(-age_sec / 3600.0)`) at line 70 — much better than the original inverse formula noted in AUDIT.
- Eviction policy (line 58-59, 150-163) prevents unbounded growth with a reasonable 200-observation cap.
- Auto-save at 60s intervals (line 22-27), `_exit_tree` (line 30-31), AND `WM_CLOSE_REQUEST` (line 33-35) — triple redundancy for data safety.
- Backward-compatible load format (line 136-145) handles both legacy arrays and new dictionaries.
- Relationship tracking (line 81-95) with clamped sentiment is a solid social model foundation.

### 4. LLM integration has proper fallbacks and rate limiting

`llm_dialogue.gd:32-44` implements a queue-based rate limiter (`MAX_CONCURRENT = 3`) that every LLM call routes through — dialogue, partner response, reflection, and agenda. Every API path has a template fallback when `OPENAI_API_KEY` is absent (e.g., `llm_dialogue.gd:147-151`). The system gracefully degrades from "LLM-powered simulation" to "template-based simulation" with zero code changes.

### 5. ContentData as single source of truth (ARCH-001)

`content_data.gd` consolidates personality text lines that were previously duplicated across `thoughts.gd` and `llm_dialogue.gd`. Both `Thoughts.get_thought()` (`thoughts.gd:6`) and `LlmDialogue._get_fallback()` (`llm_dialogue.gd:26-28`) now read from the same source. Clean fix.

### 6. UI label pool reuse

`ui.gd:251-265` — `_refresh_log()` grows the label pool incrementally and toggles visibility rather than the original destroy-and-recreate pattern. Good fix for the AUDIT Q-6 finding.

### 7. Audit trail discipline

Every code change is tagged with an issue ID (AUDIT-001, INT-005, GFX-006, etc.). TODO.md tracks phase completion. AUDIT.md and AUDIT2.md provide evidence-based reviews. This level of documentation discipline is rare in game prototypes and makes the codebase navigable.

## Code Quality Findings (Per-File)

### agent.gd — God Object (677 lines, 7+ responsibilities)

This is the most pressing quality problem. `agent.gd` currently handles:

1. **State machine** (lines 6, 214-241, 252-402)
2. **Movement & navigation** (lines 596-611)
3. **Proximity detection** (lines 616-662)
4. **Animation** (lines 529-577)
5. **UI elements** — speech bubbles, action text, emotes (lines 403-416, 580-593, 189-198)
6. **Agenda system** (lines 457-527)
7. **Personality configuration** (lines 100-131)

**Specific issues:**

- **Lines 100-131: Two match blocks for personality.** These exist despite `PersonalityProfile` being defined in `resources/personality_profile.gd` with exactly the same fields (`speed_modifier`, `seek_chance`, `approach_tendency`). The resource class is dead code. The TODO at line 99 acknowledges this debt.

- **Lines 64-172: `_ready()` is 108 lines.** It creates programmatic textures, collision shapes, name labels, speech styling, personality configuration, idle state entry, Area2D setup, AnimatedSprite2D setup, and emote icons — all in one function. This should be decomposed or moved to scene composition.

- **Lines 174-187: `_load_emote_textures()` hardcodes 7 emote paths.** If any file is missing, the emote silently fails. Fine for now, but another hardcoded data source that should move to a resource or ContentData.

- **Lines 465-500: `_get_template_agenda()` hardcodes agendas per personality.** Another match block that duplicates personality-specific data that should live in data files.

- **Line 416: `speech_timer = 4.0` hardcoded.** Combined with `interact_timer = 5.0` (line 359) and `interaction_cooldown = 5.0` (line 360), these timing constants are buried in method bodies rather than declared as class constants.

- **Line 140: `circle.radius = 80.0` hardcoded.** This should come from `AgentDefinition.detection_radius` (which exists and defaults to 80.0, but is never loaded).

### main.gd — Procedural UI spaghetti in `_ready()`

**Lines 86-155: 70 lines of manual UI reparenting.** This code:
1. Creates a `CanvasLayer` (line 87)
2. Creates a `ScreenRoot` Control (line 93-97)
3. Creates a `MarginContainer` with manual anchors (lines 100-114)
4. Reparents `UIPanel` from its scene tree parent (line 118-122)
5. Wraps `AgentList` in a `ScrollContainer` by removing and re-adding (lines 125-137)
6. Moves `UIBackground` from UIPanel to ScreenRoot (lines 140-154)

This is fragile. It depends on specific child node names, specific parent-child relationships, and specific ordering. If the scene tree changes, this code silently breaks. The correct approach is to define this layout in the `.tscn` file or in a dedicated UI scene.

**Lines 9-15: Agent data still hardcoded as Dictionary array.** The ARCHITECTURE.md Priority 1 was to extract this to `.tres` files. The `AgentDefinition` resource exists but is unused. The comment at line 8 acknowledges this.

**Lines 31-34: Zone rects hardcoded in GDScript.** These should match the TileMapLayer zones visible in `main.tscn`, but there's no programmatic link — it's manual coordination.

**Lines 305-311: Spawn positions hardcoded.** 5 fixed positions for 5 fixed agents. Adding a 6th agent requires editing two arrays in lockstep.

**Line 228: `SIM_SPEED / 3600.0` — sim time is in hours, but the constant is called `SIM_SPEED = 60.0`.** The comment says "1 real second = 1 sim minute" but the formula `delta * 60.0 / 3600.0` = `delta / 60.0` means 1 real second = 1/60 of a sim hour = 1 sim minute. Correct math, but the relationship between the constant name and its usage isn't obvious. A clearer formulation: `const SIM_MINUTES_PER_REAL_SECOND: float = 1.0` and `sim_time += delta * SIM_MINUTES_PER_REAL_SECOND / 60.0`.

**Line 216: Redundant camera lookup.** `_unhandled_input` looks up `"WorldCamera"` at line 196, then looks it up *again* at line 216 inside the drag handler (`var camera2`). The second lookup is in the same function where `camera` is already in scope; the issue is that the `if event is InputEventMouseMotion` block is outside the `if event is InputEventMouseButton` block, so `camera` isn't accessible. This should be restructured: get the camera once at the top, use it throughout.

### llm_dialogue.gd — Massive duplication in HTTP handlers

**Pattern: 6 nearly identical HTTP request/callback blocks.** These functions all follow the same structure:

1. `_do_dialogue_request` (lines 158-203)
2. `_do_partner_response_request` (lines 285-332)
3. `_do_reflection_request` (lines 349-395)
4. `_do_agenda_request` (lines 408-464)
5. `_async_rate_importance` (lines 73-116)
6. `_on_request_completed` (lines 205-257)

Each creates an `HTTPRequest`, builds a body dict, stringifies it, connects a completion callback, sends the request, and handles errors. The differences are: the prompt text, max_tokens, temperature, and what happens with the response. This should be extracted into a single `_make_llm_request(prompt, system_msg, max_tokens, temperature) -> Signal` helper.

**Lines 73-116: `_async_rate_importance` bypasses the rate limiter.** It creates its own `HTTPRequest` directly without going through `_dispatch_or_queue()`. AUDIT-017 fixed this for partner responses but missed importance rating. Under load, this can exceed the `MAX_CONCURRENT = 3` intent.

**Lines 232-257: Three identical fallback observation blocks.** Lines 214-216, 226-228, and 255-257 all record the same "couldn't find words" observation. This was noted in AUDIT Q-8 but hasn't been extracted to a helper.

### memory_service.gd — Scoring formula still duplicated

**Lines 70 and 158: Identical scoring formula.** `0.3 * exp(-age_sec / 3600.0) + 0.7 * obs["importance"]` appears in both `get_top_memories` and `_evict_lowest`. AUDIT Q-1 identified this; it hasn't been fixed. Extract to `_score_observation(obs: Dictionary, now_sec: float) -> float`.

**Lines 43-44: Real-time timestamps for a simulated world.** `Time.get_unix_time_from_system()` records wall-clock time, not sim time. If the sim runs at 60x speed (1 real second = 1 sim minute), then "recency" in memory scoring relates to how long ago the player was watching, not how long ago the agent experienced it. For a simulation, the memory system should use `sim_time` (or a monotonic sim tick counter) so that memories decay in sim-time, not real-time. This is a design mismatch that will become confusing as the time system matures.

### ui.gd — Coupling to agent internals

**Line 192: `agent._get_next_agenda_item()` — calls a private method.** The `_` prefix in GDScript conventionally means private. The UI directly reaches into the agent's internal agenda implementation. This should be exposed as a public method or property.

**Lines 198-205: Directly accesses `MemoryService.relationships` dictionary.** The UI reaches into the autoload's raw data structure rather than calling a public API method. If the data structure changes, the UI breaks. Consider adding `MemoryService.get_average_sentiment(agent_name) -> float`.

**Line 174: `_process` polls every agent every frame.** With 5 agents this is negligible. With 50 agents, this creates per-frame string formatting, dictionary lookups, and memory scoring (via `get_top_memories` at line 209). Consider polling at a lower rate (e.g., every 0.5 seconds via timer) or using signals for state changes.

### thoughts.gd — Minimal but viable

At 9 lines, this is the cleanest file. It does one thing: returns a random thought from ContentData. The only concern is that it's a thin wrapper around `ContentData.PERSONALITY_LINES` — it could be inlined. But having it as a named autoload provides a clear extension point for future LLM-powered idle thoughts, so keeping it is defensible.

### content_data.gd — Good but should grow

11 lines, single dictionary. This is the right pattern. As the project grows, this should hold zone descriptions, backstory fragments, time-of-day flavor text, etc. Consider whether this should remain a `const Dictionary` or become a Resource loaded from `.tres` for modding support.

### resources/agent_definition.gd and resources/personality_profile.gd — Dead code

Both files are well-designed resource definitions that exactly match what ARCHITECTURE.md proposed. Neither is loaded, referenced, or instantiated anywhere in the runtime code. No `.tres` files exist for them. They represent intent that was never followed through.

### Trade-off assessment:

The project chose **breadth over depth** — implementing many features (agenda, reflection, relationships, lighting, camera, sprites) rather than first solidifying the data-driven foundation. This is the classic prototype trap: each new feature is fun to build and visibly impressive, but each one adds more hardcoded data and more responsibilities to `agent.gd`.

The architecture doc explicitly warned against this: "This is the foundation everything else builds on. Until agent data lives in .tres files instead of hardcoded arrays and match statements, nothing else can be data-driven." That warning went unheeded.

Was this the wrong call? For a prototype, **probably not**. The breadth exploration validated which features matter (reflections are cool, agendas add narrative, lighting sells the mood). But the technical debt is now significant, and the next phase should be consolidation, not more features.

## Top Risks / Things to Fix Before Scaling

### Risk 1: agent.gd will become unmaintainable

At 677 lines with 7 responsibilities, this file is already at the edge of comprehensibility. Every new feature (emotions, routines, multi-turn dialogue, object interactions) will add more state and more methods here. By 1000 lines it'll be impenetrable.

**Mitigation:** Extract into components:
- `AgentStateMachine` (or use Godot's built-in `StateMachine` pattern)
- `AgentMovement` (navigation, movement helpers)
- `AgentSocial` (proximity detection, interaction logic, seek/flee)
- `AgentAgenda` (daily planning, agenda following)
- Animation handling should live in the scene tree (AnimatedSprite2D configured in .tscn)

### Risk 2: Wall-clock memory timestamps vs sim-time

`memory_service.gd:43` uses `Time.get_unix_time_from_system()`. Memory recency decays based on real seconds, not sim-time. At 60x speed, a 1-minute real-world session covers 1 sim-hour, but memories decay as if only 60 real seconds passed. If the sim is paused, memories still decay. If the sim speed changes, decay rate doesn't adjust.

**Impact:** As the time system becomes more important (day cycles, schedules, temporal reasoning), this mismatch will create incoherent agent behavior. An agent's "morning memories" will have the same recency score as "afternoon memories" if they happened 2 real minutes apart.

**Mitigation:** Use a sim-time epoch counter. Pass `sim_time` to MemoryService, or maintain a monotonic sim-tick counter that respects pause state.

### Risk 3: Closure captures in HTTP callbacks hold node references

Throughout `llm_dialogue.gd`, HTTP callbacks capture `agent` and `other` CharacterBody2D references in closures (e.g., lines 189-194, 315-327). These closures fire after async HTTP responses — potentially seconds later. If nodes are freed between request and response, `is_instance_valid()` checks at callback entry (line 206, 271, 286, 323) catch this.

But: the closures themselves keep the Variant references alive, preventing Godot from fully freeing the nodes until the closure is garbage-collected. With queued requests waiting for rate-limit slots, this could hold stale references for extended periods.

**Mitigation:** Store `instance_id` instead of node references in closures, and look up via `instance_from_id()` in the callback.

### Risk 4: No error recovery in memory loading

`memory_service.gd:119-148` — if a JSON file is corrupt (partial write from a crash), `json.parse()` fails silently and the agent starts with zero memories. There's no error logging, no backup, no recovery attempt.

**Mitigation:** Add `push_error()` on parse failure. Consider writing to a temp file and renaming atomically in `save_memories()`. Keep one backup copy.

### Risk 5: Hardcoded spawn positions and agent data can't scale

`main.gd:9-15` (5 agents) and `main.gd:305-311` (5 positions) are tightly coupled arrays. Adding agent 6 requires editing both in lockstep. The positions are hardcoded pixel coordinates with no relation to zone_rects.

**Mitigation:** Generate spawn positions from zone_rects (random point within a random zone). Load agent data from resources or a JSON config file.

### Risk 6: UI per-frame polling doesn't scale

`ui.gd:174-225` polls every agent every frame. Each poll calls `MemoryService.get_top_memories()` (line 209), which iterates all observations and sorts them — O(n log n) per agent per frame. With 5 agents and 200 observations each, that's 5 sorts of 200-element arrays per frame. At 20 agents, it's 20 sorts per frame. At 60fps, that's 1200 sorts/second.

**Mitigation:** Poll on a timer (every 0.5s) or use a dirty flag that agents set when their state changes.

## Recommended Next Steps (Prioritized)

### P0 — Consolidation (do before any new features)

1. **Wire in PersonalityProfile resources.** Create 5 `.tres` files (one per personality), load them in `agent.gd:_ready()`, delete the match blocks at lines 100-131. Estimated effort: 1 session. This is the single highest-leverage change.

3. **Extract LLM HTTP helper.** Create `_make_api_request(messages: Array, max_tokens: int, temperature: float, callback: Callable)` in `llm_dialogue.gd`. Refactor the 5 duplicate request builders to use it. This will halve the file size and eliminate the maintenance burden of keeping 5 blocks in sync.

4. **Route `_async_rate_importance` through the rate limiter.** It's the one remaining code path that bypasses `_dispatch_or_queue()`. One-line fix to wrap the call.

5. **Declare magic numbers as constants.** At minimum in `agent.gd`: `DETECTION_RADIUS`, `INTERACT_DISTANCE`, `SEEK_ABANDON_DISTANCE`, `INTERACT_DURATION`, `INTERACTION_COOLDOWN`, `SPEECH_DURATION`, `FLEE_DISTANCE`, `FLEE_COOLDOWN`. Grep for bare numeric literals and replace.

6. **Move UI setup out of main.gd.** The CanvasLayer/reparenting code (main.gd:86-155) should either live in the `.tscn` scene definition or in a separate `ui_setup.gd` script. main.gd should be about game logic, not UI layout.

7. **Switch memory timestamps to sim-time.** Add a `sim_time` parameter to `add_observation()` and store it alongside or instead of wall-clock time. Update the scoring formula accordingly.

9. **Throttle UI polling.** Add a 0.5s timer to `ui.gd` instead of per-frame `_process`. Or switch to signal-driven updates.

12. **Create AgentDefinition .tres files.** One per agent, containing name, personality, color, backstory. Load in main.gd instead of the Dictionary array. This completes ARCHITECTURE.md Priority 1.
